# 前端验证与模块说明

这份文档用于解释当前 CareerPilot 前端每个主要区域的含义，以及在页面上操作后应该看到什么结果。

## 1. 顶部文本框输入得到的是什么？

首页中间的圆角文本框是 `Run goal`，也就是“本次 Agent 运行目标”。

默认文案：

```text
为 AI Agent 实习岗位生成可追踪运行计划，保留人工审批点和成本记录。
```

点击右侧发送按钮后，前端调用：

```http
POST /api/runs
```

它得到的不是普通聊天回答，而是一个 `AgentRun` 运行记录。

返回内容会进入下方 `Active Trace` 区域，核心字段包括：

- `run_id`：这次运行的唯一 ID。
- `state`：运行状态，例如 `CREATED`、`RUNNING`、`WAITING_APPROVAL`、`COMPLETED`。
- `steps`：Agent 执行过哪些步骤。
- `events`：运行过程中的事件日志。
- `checkpoints`：中间结果快照。
- `costs`：LLM 调用成本记录。

所以，顶部文本框目前不是 ChatGPT 式问答框。它更像“启动一个可追踪 Agent 任务”的命令入口。

## 2. W2 文本框输入得到的是什么？

W2 区域有两个文本框：

- `Resume`：简历文本。
- `Job description`：岗位 JD 文本。

点击 `Parse resume` 后，前端调用：

```http
POST /api/parsers/resume
```

输出 `ResumeProfile`：

- `education`：教育经历。
- `skills`：技能。
- `projects`：项目经历。
- `experiences`：实习或工作经历。
- `keywords`：后续匹配使用的关键词。
- `evidence`：字段来自原文的哪一句。
- `metadata`：解析来源、模型名、是否 dry-run、是否修复 JSON。

点击 `Parse JD` 后，前端调用：

```http
POST /api/parsers/job
```

输出 `JobProfile`：

- `company`：公司。
- `title`：岗位名称。
- `hard_requirements`：硬性要求。
- `nice_to_have`：加分项。
- `responsibilities`：岗位职责。
- `tech_keywords`：技术关键词。
- `hidden_keywords`：隐性能力关键词。
- `evidence`：字段来自 JD 原文的哪一句。

W2 的意义是把非结构化文本变成后续 Agent 可使用的结构化数据。

## 3. W2 / W3 / W4 分别负责什么？

### W2 Parser：结构化入口

W2 负责把简历和 JD 解析成结构化 profile。

它回答的问题是：

```text
用户真实有什么经历？
岗位真实要求什么能力？
这些字段分别来自原文哪里？
```

它和项目主题的关系：

CareerPilot 不是随便生成求职材料，而是 evidence-locked workflow。W2 是证据入口，没有 W2，后面的匹配、改写、审批都没有可信输入。

### W3 LoopEngine：可追踪运行框架

W3 负责把一次 Agent 工作流组织成：

```text
Plan → Execute → Verify → Reflect → Human Approval → Commit
```

它回答的问题是：

```text
Agent 做了哪些步骤？
失败后能不能恢复？
关键节点有没有人工审批？
每一步有没有 event / checkpoint？
```

它和项目主题的关系：

CareerPilot 的核心不是单次生成，而是“可追踪、可复盘、可审批”的 Agent 工作流。W3 是运行时骨架。

### W4 MatchAgent：可解释岗位匹配

W4 使用 W2 的 `ResumeProfile` 和 `JobProfile`，生成：

- `overall_score`：总体匹配度。
- `score_breakdown`：硬性要求、加分项、职责、关键词分项分。
- `evidence_mapping`：岗位要求对应哪些简历证据。
- `gaps`：当前简历没有覆盖好的缺口。
- `priority_ranking`：下一步改简历的优先级。

它回答的问题是：

```text
这份简历和这个岗位匹配到什么程度？
分数为什么是这个结果？
哪些内容有证据支撑？
哪些内容不能直接写，需要补充真实证据？
```

它和项目主题的关系：

W4 把“我适不适合这个岗位”变成可解释的证据链，而不是让模型凭感觉打分。

## 4. W5 ResumeRewriteAgent 当前负责什么？

W5 使用 W2 的结构化 profile 和 W4 的匹配结果，生成可审批的简历改写草稿：

- `headline`：面向目标岗位的标题。
- `target_keywords`：应该优先呈现的关键词。
- `changes`：改写 diff。
- `evidence`：每条改写依据哪些原始证据。
- `risk_warnings`：不能直接写进简历的缺口和风险。
- `approval_status`：是否等待人工审批。

当前 W5 的安全规则：

```text
有证据的内容可以改写表达。
没有证据的内容只能提示风险，不能伪造成经历。
审批完成前不能导出 PDF。
```

它和项目主题的关系：

W5 是从“分析”走向“生成”的第一步，但仍然保持 human-in-the-loop。也就是说，CareerPilot 可以帮你写得更贴合岗位，但不能替你编造经历。

## 5. 页面验证路径与预期结果

### 验证 W2

操作：

1. 在 `Resume` 文本框保留或输入简历。
2. 点击 `Parse resume`。
3. 在 `Job description` 文本框保留或输入 JD。
4. 点击 `Parse JD`。

预期：

- `Resume Profile` 卡片展示技能、项目、教育等数量。
- `Job Profile` 卡片展示公司、岗位、硬性要求、加分项等数量。
- `Active Trace` 显示 ParserAgent 的 step、event、checkpoint。

### 验证 W3

操作：

1. 顶部文本框保留一个运行目标。
2. W2 区域至少保留一段简历或 JD。
3. 点击 `Start loop`。

预期：

- LoopEngine 状态进入 `WAITING_APPROVAL`。
- `Active Trace` 展示 plan、execute、verify、reflect、human_approval 等步骤。
- 输入审批备注后点击 `Approve`，运行进入 `COMPLETED`。

### 验证 W4

操作：

1. 先完成 `Parse resume`。
2. 再完成 `Parse JD`。
3. 点击 `Run match`。

预期：

- W4 面板出现匹配分数。
- 出现 matched / missing keywords。
- `Top gaps` 展示缺口。
- `Priorities` 展示 P0 / P1 / P2 优先级。
- `Active Trace` 出现 `MatchAgent` 的 `match` step。

### 验证 W5

操作：

1. 先完成 W2 简历解析和 JD 解析。
2. 点击 W4 的 `Run match`。
3. 点击 W5 的 `Create draft`。
4. 审查 W5 面板中的 diff、evidence 和 risk warnings。
5. 输入审批备注，点击 `Approve draft`。
6. 审批完成后点击 `Export PDF`。

预期：

- 未完成 W4 前，W5 不允许生成草稿。
- 生成草稿后，运行状态进入 `WAITING_APPROVAL`。
- W5 面板显示改写建议和风险提示。
- 未审批前导出 PDF 会被后端拒绝。
- 审批后运行进入 `COMPLETED`，`Export PDF` 可下载 PDF。

### 验证 W6

操作：

1. 在岗位收集区域粘贴公开 JD 文本、HTML 片段或 URL。
2. 点击收集岗位。

预期：

- 系统生成 JobSnapshot。
- 可看到来源类型、文本 hash、截图状态和安全检查结果。
- JD 会进入结构化解析，供 W4/W5/W7/W8 使用。

### 验证 W7

操作：

1. 完成 W2、W4、W5。
2. 点击生成面试包。

预期：

- 页面展示预测问题、项目追问、回答框架、需要补强的知识点和真实性提醒。
- 问题应围绕真实项目经历和岗位要求，而不是空泛脑筋急转弯。
- STAR 不是故事模板，而是按背景、任务、行动、结果组织真实项目回答。

### 验证 W8

操作：

1. 完成 W2-W7。
2. 点击保存投递记录。
3. 修改投递状态或添加面试反馈。

预期：

- 页面展示投递状态、长期记忆、下一步任务和最近投递。
- CRM 只保存记录和任务，不会自动替用户投递。

### 验证 W9

操作：

1. 完成 W2-W8。
2. 点击运行质量评测。
3. 点击打开 HTML 报告。

预期：

- 页面展示 QualityGate 分数、PASS/WARN/BLOCK、通过/警告/失败规则。
- HTML 报告可单独打开。
- 若存在无证据改写或缺失章节，会出现 warning/block。

### 验证 W10

操作：

1. 后端运行后访问 `GET /api/production/readiness`。
2. 访问 `GET /api/production/cost-summary`。
3. 可选执行 `scripts/demo_w10.sh`。

预期：

- readiness 显示 LLM/Judge 是否配置、鉴权是否开启、限流值。
- cost summary 显示 run 数、成本记录、token 和按模型聚合的成本。
- demo 脚本能完成 health、readiness、评测和成本汇总 smoke test。

## 6. 当前前端模块与主题关系总览

| 前端模块 | 对应 Week | 输入 | 输出 | 与主题的关系 |
|---|---:|---|---|---|
| 顶部 Run goal | W1 | Agent 运行目标 | Run Trace | 创建可追踪运行 |
| Structured intake | W2 | 简历 / JD 文本 | ResumeProfile / JobProfile | 证据入口 |
| Plan · Execute · Verify | W3 | 目标 + 简历/JD | 可恢复工作流 | Agent 运行骨架 |
| Fit score · Evidence · Gaps | W4 | ResumeProfile + JobProfile | 匹配分、证据、缺口 | 可解释匹配 |
| Rewrite draft · Diff · Approval | W5 | W2 + W4 结果 | 改写草稿、审批、PDF | 证据锁定生成 |
| Job Collector | W6 | URL / HTML / 文本 | JobSnapshot + JobProfile | 岗位来源留证 |
| Interview Pack | W7 | W2/W4/W5 | 面试题、STAR 框架、补强项 | 投递前准备 |
| Application CRM | W8 | W2-W7 产物 | 投递记录、记忆、任务 | 长期求职闭环 |
| QualityGate | W9 | W2-W8 产物 | 质量分、HTML 报告 | 风险拦截 |
| Production Panel / API | W10 | 环境配置 + runs | readiness、cost summary | 可演示、可部署 |
| Active Trace | W1-W10 | run_id | steps/events/checkpoints/costs | 可复盘与可审计 |
